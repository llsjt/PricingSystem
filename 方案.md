# 从数据库轮询式任务队列改为基于 RabbitMQ 的异步任务系统

> **目标定位：** 这是一次**架构演进**，不是生产上线改造。目的是把当前"数据库兼职任务队列 + 持续轮询"的链路换成"MySQL 做真相源 + RabbitMQ 做异步传输"的稳定结构，解决 **Worker 崩溃难恢复**、**晚到写回覆盖新结果**、**SSE 持续查库** 三个痛点。单实例部署、演示级流量。

**Goal:** 去掉 Python 对数据库的认领轮询，改为 RabbitMQ 消费 + execution fencing；去掉 Java SSE 对数据库的持续轮询（独立阶段，可选）。`pricing_task / agent_run_log / pricing_result` 仍然是业务真相源。

**Tech Stack:** Spring Boot 3.2 + Spring AMQP + RabbitMQ 3.13 + FastAPI + aio-pika + MySQL 8

---

## 1. 现状与痛点

当前链路：

1. Java `DecisionTaskServiceImpl.startTask()` 插入 `pricing_task`（状态 `PENDING`），随后通过 `PythonDispatchClient` 同步 HTTP 调 Python `/internal/tasks/dispatch`。
2. Python `task_queue_service._worker_loop()` 同时用 `claim_next_dispatchable()` 持续轮询数据库认领 `QUEUED/RETRYING` 任务——作为 HTTP 失败时的兜底。
3. Java `PricingTaskStreamService.streamLoop()` 用 `while(true) + Thread.sleep` 持续查 `pricing_task / agent_run_log / pricing_result`，把变化作为 SSE 推给前端。

三个痛点：

- **Worker 恢复能力弱**：Python Worker 执行中崩溃只能靠"DB 里看到长时间 RUNNING 再改回 RETRYING"这种状态修复，没有消息层的 ack / requeue 保护。
- **晚到写回风险**：同一个 task 被两次 attempt 执行（假死后 RabbitMQ 超时重投、或现在的 HTTP + DB 轮询双路径），旧 attempt 的写回可能覆盖新 attempt 的结果。
- **SSE 持续查库**：业务真相源同时承担实时事件源，每个 SSE 连接都在轮询同一批表。

## 2. 目标架构

```
Frontend
  → Java Backend
    ↕ MySQL: pricing_task / agent_run_log / pricing_result   (真相源，不变)
    ↕ RabbitMQ:
        pricing.task.dispatch.exchange  (Java → Python)
        pricing.task.progress.exchange  (Python → Java，阶段 3 启用)

Java 创建任务（三步独立小事务，见 §7.3）：
  1. 插入 pricing_task，status=PENDING（提交）
  2. publish dispatch 消息
  3. UPDATE status=QUEUED（提交）

Python Worker：
  1. 消费 dispatch queue
  2. 用 asyncio.to_thread 包裹同步 repo 调用（不阻塞事件循环）
  3. CAS 占用：把 pricing_task.current_execution_id 从 NULL 原子改成本次 execution_id
  4. 抢不到就 ack 放弃（已被其他 attempt 占走、或已终态）
  5. 执行业务，所有写回都带 owner guard：UPDATE 带 WHERE current_execution_id = ? + 终态守卫，日志 INSERT 带 WHERE EXISTS，结果写入用 guarded upsert
  6. 每次关键状态变更 publish 一条 progress 消息（阶段 3 启用；阶段 2 空操作）
  7. 成功或不可恢复失败都 ack；可恢复异常 nack 让 RabbitMQ requeue

Java SSE（阶段 3，可选）：
  1. 建连时查一次数据库快照回放
  2. 注册 SseEmitter 到内存 Map<taskId, List<SseEmitter>>
  3. 消费 progress queue 按 taskId 扇出（单实例假设）
```

### 设计原则

- **MySQL 是真相源**，RabbitMQ 只传输。进度消息丢了不影响业务正确性，SSE 重连时查库补齐。
- **事务边界与消息边界分离**：`startTask` **不再整块 `@Transactional`**。三步各自是独立小事务——这样能彻底避免"publish 在 DB commit 之前先到 broker"导致 Python 查不到任务的幽灵消息问题。
- **两段式任务状态**：insert 时 `PENDING`，publish 成功后 UPDATE 为 `QUEUED`。Python 消费者对 `PENDING/QUEUED/RUNNING/RETRYING` 都视为可处理状态（`PENDING` 可能出现在 Java 已 publish 但还没来得及 UPDATE QUEUED 的一小段窗口），终态直接 ack。
- **execution fencing + 终态守卫**：每次 Worker 消费消息都生成一个 UUID `execution_id`。普通消息只能在 `current_execution_id IS NULL` 时抢占；RabbitMQ 标记为 `redelivered` 的消息允许从旧 `execution_id` 接管并把 `consumer_retry_count + 1`。所有 UPDATE 写回都带 `WHERE current_execution_id = ?`，并额外保护 `task_status NOT IN ('COMPLETED','FAILED','CANCELLED','MANUAL_REVIEW')`；所有 INSERT/UPSERT 写回用 owner guard。晚到的 attempt、用户已取消的任务、已经终态的任务，受影响行数都应为 0。这是防"双跑 + 晚写覆盖 + 取消后被覆盖"的核心防线。
- **消费者幂等基于业务状态 + fencing**：进入 `on_message` 先查 `pricing_task.task_status`，终态直接 ack；非终态尝试 CAS；抢到才执行。不建独立去重表。
- **单实例部署假设**：一个 Java 进程 + 一个 Python 进程 + 一个 RabbitMQ 实例。SSE 扇出用内存 Map 即可，不做跨节点 fanout。
- **publish 必须有 broker 确认**：不引入 Outbox 表，但不能只以 `convertAndSend` 未抛异常作为入队成功。Java 开启 publisher confirm + mandatory returns；只有 broker confirm ack 且没有 unroutable return，才把 task 从 `PENDING` 转 `QUEUED`。发送异常、confirm nack、routing 失败都视为派发失败，独立事务把 task 置为 `FAILED` 并抛异常。
- **PENDING 兜底采用补发而不是直接失败**：Java 启动后扫描长时间停留在 `PENDING` 的任务，重新 publish dispatch 并转 `QUEUED`（见 §15）。重复消息由 fencing 消化，避免"publish 已成功但 Java 死在 UPDATE QUEUED 前"的任务被误判失败。
- **SSE 合约保持前端兼容**：阶段 3 的 RabbitMQ progress 只是 Java SSE 的唤醒/通知源，不能直接把 `TASK_STARTED` 等内部事件透传给浏览器。Java 必须继续向前端发送当前约定的 `task_started / agent_card / task_completed / task_failed` payload；如果 progress payload 不含完整卡片内容，Java 收到事件后查一次 DB/log/result 组装现有 SSE 消息。

## 3. RabbitMQ 拓扑（极简）

**派发链路（阶段 1 就声明）**
- Exchange: `pricing.task.dispatch.exchange`（direct，durable）
- Queue: `pricing.task.dispatch.queue`（durable）
- Routing Key: `pricing.task.dispatch`

**进度链路（阶段 3 才声明与绑定）**
- Exchange: `pricing.task.progress.exchange`（direct，durable）
- Queue: `pricing.task.progress.java.queue`（durable）
- Routing Key: `pricing.task.progress`

两个 queue 都开持久化，消息也用 `MessageDeliveryMode.PERSISTENT`。**不声明** retry exchange、延迟队列、DLX——重试由应用层基于 `consumer_retry_count` 统一处理（见 §6.3），超限直接 mark_failed + ack。

**阶段 3 skip 保护**：Python 侧用配置开关 `PROGRESS_PUBLISH_ENABLED`（默认 `false`）控制是否 publish progress。只有阶段 3 上线时才把它改为 `true` 并声明对应 exchange/queue。这样阶段 2 独立上线时，不存在"progress 消息进队无人消费堆积"的问题。

## 4. 消息模型

### 4.1 Dispatch 消息（Java → Python）

```json
{
  "eventId": "uuid",
  "taskId": 123,
  "traceId": "trace-xxx",
  "occurredAt": "2026-04-21T04:39:00.123Z"
}
```

**关键决策**：消息只带 `taskId` 和 `traceId`。Java 创建任务时把当前用户的 `llm_api_key_enc / llm_base_url / llm_model` 复制到 `pricing_task`；Python 消费时只从 `pricing_task` 读取任务参数和 LLM 配置，并在 Python 端用新增的 AES-GCM 工具解密 `llm_api_key_enc`（见 §6.6）。消息体不携带任何密钥。

好处：LLM 密钥不走 RabbitMQ；消息体小；任务参数永远以 DB 为准，不会被过时的消息体污染。

**时间/字符串约定**：
- `occurredAt` 一律 UTC + ISO-8601 + 毫秒精度 + `Z` 后缀。Java 用 `DateTimeFormatter.ISO_INSTANT` + `Instant.now()`；Python 用 `datetime.now(timezone.utc).isoformat(timespec="milliseconds").replace("+00:00", "Z")`。
- 所有字符串字段 UTF-8；`traceId` 使用 32 位 hex（去掉 `-` 的 UUID）。

### 4.2 Progress 消息（Python → Java，阶段 3 启用）

```json
{
  "eventId": "uuid",
  "eventType": "TASK_STARTED | AGENT_CARD_RUNNING | AGENT_CARD_COMPLETED | TASK_COMPLETED | TASK_FAILED | TASK_MANUAL_REVIEW",
  "taskId": 123,
  "executionId": "uuid",
  "traceId": "trace-xxx",
  "payload": { ... },
  "occurredAt": "2026-04-21T04:40:00.456Z"
}
```

`executionId` 带上用于 Java 识别事件所属的 attempt（比如 attempt 1 晚到的 `TASK_STARTED` 应该被 Java 忽略）。**Progress 消息是服务端内部消息，不等同于浏览器 SSE payload**。Java 消费后必须转换为现有前端合约：

- `task_started`
- `agent_card`
- `task_completed`
- `task_failed`

当前前端 `PricingLab.vue` 依赖 `agent_card.agentCode / displayOrder / stage / card.thinking / card.evidence / card.suggestion` 等字段，不能直接接收 `TASK_STARTED / AGENT_CARD_COMPLETED` 这类内部事件名。

**每个 eventType 的 payload 字段**：

| eventType | payload 字段 |
|-----------|--------------|
| `TASK_STARTED` | `{}`（空对象） |
| `AGENT_CARD_RUNNING` | `{ "agentName": "DataAnalysis", "runAttempt": 1 }` |
| `AGENT_CARD_COMPLETED` | `{ "agentName": "DataAnalysis", "runAttempt": 1, "runStatus": "success\|failed", "durationMs": 4321 }` |
| `TASK_COMPLETED` | `{ "finalPrice": "99.00", "confidence": 0.87 }` |
| `TASK_FAILED` | `{ "reason": "..."（≤255 字符） }` |
| `TASK_MANUAL_REVIEW` | `{ "reason": "..."（≤255 字符） }` |

Java 侧 `TaskProgressEvent` DTO 仅反序列化顶层 + `payload` 透明 `Map<String,Object>`。收到事件后推荐走"轻通知 + DB 补齐"模式：

1. 校验 `taskId` 对应的 `current_execution_id` 是否仍等于事件里的 `executionId`；不相等说明是旧 attempt 的晚到事件，直接 ack 丢弃。
2. 根据 `eventType` 判断要补发哪类 SSE 消息。
3. 查询一次 `pricing_task / agent_run_log / pricing_result`，复用当前 `PricingTaskStreamService.toAgentCard / buildResultPayload / baseMessage` 的组装逻辑。
4. 向浏览器发送当前前端已经支持的 SSE payload。

如果后续选择让 Python progress 直接携带完整 SSE payload，也必须保持字段与当前 `frontend/src/api/decision.ts` 的 `PricingTaskStreamMessage` 类型一致；不允许把内部 progress schema 直接暴露给前端。

### 6.3 消费入口骨架

```python
import asyncio
import json
import uuid
from aio_pika.abc import AbstractIncomingMessage

TERMINAL_STATES = {"COMPLETED", "FAILED", "CANCELLED", "MANUAL_REVIEW"}
FAILURE_REASON_MAX_LEN = 255


def _truncate(msg: str | None, limit: int = FAILURE_REASON_MAX_LEN) -> str:
    if msg is None:
        return ""
    return msg if len(msg) <= limit else msg[: limit - 3] + "..."


async def on_message(message: AbstractIncomingMessage) -> None:
    try:
        payload = json.loads(message.body)
        task_id = int(payload["taskId"])
    except Exception:
        await message.ack()  # 消息格式坏，直接丢弃避免死循环
        return

    # 所有 repo 方法都是同步 SQLAlchemy；用 to_thread 避免阻塞事件循环
    task = await asyncio.to_thread(repo.get_task, task_id)
    if task is None:
        # 极罕见：Java insert 事务未提交 or 已回滚。ack 丢掉；§15 孤儿扫描会收尾。
        await message.ack()
        return
    if task.task_status in TERMINAL_STATES:
        await message.ack()
        return

    retry_exhausted = (task.consumer_retry_count or 0) >= settings.worker_max_retry
    has_owner = bool(task.current_execution_id)
    if retry_exhausted:
        # 有 owner 且不是 redelivery，说明只是重复 dispatch 消息，不能误杀正在执行的任务。
        if (not has_owner) or bool(message.redelivered):
            await asyncio.to_thread(
                repo.mark_failed_force, task_id, _truncate("超过最大消费重试次数")
            )
            await progress.publish("TASK_FAILED", task_id, None,
                                   {"reason": "超过最大消费重试次数"})
        await message.ack()
        return

    # CAS 占用。RabbitMQ redelivery 表示上一条 unacked 消息的 consumer 已断开，
    # 可安全接管旧 execution；晚到旧 attempt 的写回会被 fencing 拒绝。
    execution_id = str(uuid.uuid4())
    acquired = await asyncio.to_thread(
        repo.acquire_execution,
        task_id,
        execution_id,
        allow_reclaim=bool(message.redelivered),
        max_retry=settings.worker_max_retry,
    )
    if not acquired:
        await message.ack()
        return

    try:
        # dispatch_service 内部继续用 to_thread 包裹所有同步 DB 调用
        await dispatch_service.run_task(task_id, execution_id)
        await message.ack()
    except RecoverableError as e:
        await asyncio.to_thread(
            repo.increment_consumer_retry_and_release,
            task_id, execution_id, _truncate(str(e))
        )
        retry_count = int((task.consumer_retry_count or 0) + 1)
        backoff = min(2 ** retry_count, settings.worker_retry_backoff_max_seconds)
        await asyncio.sleep(backoff)  # 不引入延迟队列时，用应用层退避避免热循环
        await message.nack(requeue=True)
    except Exception as e:
        await asyncio.to_thread(
            repo.mark_failed_if_owner, task_id, execution_id, _truncate(str(e))
        )
        await progress.publish("TASK_FAILED", task_id, execution_id,
                               {"reason": _truncate(str(e))})
        await message.ack()
```

要点：
- `prefetch=1` + `to_thread`：同一时刻最多一条消息在执行；同步 repo/LLM 调用不阻塞 aio-pika 心跳。
- 所有写入失败原因前统一 `_truncate` 到 255 字符，和 Java `truncate(e.getMessage(), 255)` 对齐，避免 `Data too long for column` 导致死循环。

### 6.4 关键 SQL

```sql
-- acquire_execution：普通消息只认 current_execution_id IS NULL；
-- RabbitMQ redelivered 消息允许接管旧 execution，并计入消费重试。
UPDATE pricing_task
   SET current_execution_id = :new_id,
       task_status = 'RUNNING',
       started_at = NOW(),
       completed_at = NULL,
       consumer_retry_count = consumer_retry_count + CASE
           WHEN current_execution_id IS NULL THEN 0 ELSE 1
       END,
       updated_at = NOW()
 WHERE id = :task_id
   AND task_status NOT IN ('COMPLETED','FAILED','CANCELLED','MANUAL_REVIEW')
   AND consumer_retry_count < :max_retry
   AND (
       current_execution_id IS NULL
       OR (:allow_reclaim = 1 AND current_execution_id IS NOT NULL)
   );
-- 返回影响行数 == 1 表示占用成功

-- UPDATE 写回（任务状态切换 / 写 final_price / 写 failure_reason）
UPDATE pricing_task
   SET task_status = :new_status, ...
 WHERE id = :task_id
   AND current_execution_id = :my_execution_id
   AND task_status NOT IN ('COMPLETED','FAILED','CANCELLED','MANUAL_REVIEW');

-- INSERT 写回（agent_run_log）带 EXISTS 守卫：
--   晚到 attempt 的 execution_id 早已不再是当前值，EXISTS 查不到，受影响行数 = 0
INSERT INTO agent_run_log (task_id, execution_id, role_name, stage, ...)
SELECT :task_id, :my_execution_id, :role_name, :stage, ...
  FROM pricing_task
 WHERE id = :task_id
   AND current_execution_id = :my_execution_id
   AND task_status NOT IN ('COMPLETED','FAILED','CANCELLED','MANUAL_REVIEW');

-- pricing_result 有 uk_pricing_result_task_id，必须用 guarded upsert。
-- 旧 attempt 如果已写过 result 但没来得及把 task 置终态，新 attempt 仍能覆盖；
-- 晚到旧 attempt 则因 current_execution_id 不匹配无法覆盖。
INSERT INTO pricing_result (task_id, execution_id, final_price, ...)
SELECT :task_id, :my_execution_id, :final_price, ...
  FROM pricing_task
 WHERE id = :task_id
   AND current_execution_id = :my_execution_id
   AND task_status NOT IN ('COMPLETED','FAILED','CANCELLED','MANUAL_REVIEW')
ON DUPLICATE KEY UPDATE
    execution_id = IF(
        (SELECT current_execution_id FROM pricing_task WHERE id = :task_id)
            = VALUES(execution_id)
        AND (SELECT task_status FROM pricing_task WHERE id = :task_id)
            NOT IN ('COMPLETED','FAILED','CANCELLED','MANUAL_REVIEW'),
        VALUES(execution_id),
        execution_id
    ),
    final_price = IF(
        (SELECT current_execution_id FROM pricing_task WHERE id = :task_id)
            = VALUES(execution_id)
        AND (SELECT task_status FROM pricing_task WHERE id = :task_id)
            NOT IN ('COMPLETED','FAILED','CANCELLED','MANUAL_REVIEW'),
        VALUES(final_price),
        final_price
    );

-- increment_consumer_retry_and_release：释放占用，让下次 requeue 能重新获取
UPDATE pricing_task
   SET consumer_retry_count = consumer_retry_count + 1,
       current_execution_id = NULL,
       task_status = 'RETRYING',
       started_at = NULL,
       completed_at = NULL,
       failure_reason = :reason
 WHERE id = :task_id
   AND current_execution_id = :my_execution_id
   AND task_status NOT IN ('COMPLETED','FAILED','CANCELLED','MANUAL_REVIEW');

-- mark_failed_if_owner：终态失败，保留 execution 痕迹用于排查
UPDATE pricing_task
   SET task_status = 'FAILED',
       failure_reason = :reason,
       completed_at = NOW()
 WHERE id = :task_id
   AND current_execution_id = :my_execution_id
   AND task_status NOT IN ('COMPLETED','FAILED','CANCELLED','MANUAL_REVIEW');

-- cancel_if_running：用户取消任务时主动废弃当前 owner，
-- 防止仍在运行的旧 worker 后续把 CANCELLED 覆盖为完成态。
UPDATE pricing_task
   SET task_status = 'CANCELLED',
       failure_reason = '任务已取消',
       current_execution_id = NULL,
       completed_at = NOW(),
       updated_at = NOW()
 WHERE id = :task_id
   AND task_status IN ('PENDING','QUEUED','RUNNING','RETRYING');

-- mark_failed_force：超重试上限时使用，不带 fencing（因为此时可能并未持有 execution）
UPDATE pricing_task
   SET task_status = 'FAILED',
       failure_reason = :reason,
       completed_at = NOW()
 WHERE id = :task_id
   AND task_status NOT IN ('COMPLETED','FAILED','CANCELLED','MANUAL_REVIEW');
```

关键点：
- **CAS 真互斥**：普通消息只用 `IS NULL` 抢占；只有 RabbitMQ `redelivered` 消息可以接管旧 `execution_id`。`!= :new_id` 因 UUID 永远不相等会退化为"永远抢得到"，不能使用。
- **INSERT / UPSERT 也受 fencing 保护**：`agent_run_log` 用 `INSERT ... SELECT ... WHERE`；`pricing_result` 因存在唯一键，必须用 guarded upsert，避免旧 attempt 覆盖新结果。
- **终态不可覆盖**：所有 owner 写回除了匹配 `current_execution_id`，还要排除 `COMPLETED/FAILED/CANCELLED/MANUAL_REVIEW`。这能防止用户取消后，仍在运行的 worker 把任务重新写成完成态。
- **重试计数在 DB，且用独立列**：`consumer_retry_count` 和历史 `retry_count`（业务层重试）语义不共用一份额度。
- **可恢复 / 不可恢复分开**：网络抖动、LLM 瞬时错误走 sleep backoff + nack+requeue（释放 execution_id）；参数损坏、任务已删除、LLM 密钥缺失等走 ack + mark_failed。

### 6.5 progress 发布的时机

`dispatch_service.run_task(task_id, execution_id)` 里在下列 6 个点调用 `progress.publish(event_type, task_id, execution_id, payload)`：
1. 任务切到 `RUNNING`
2. 每张 Agent 卡开始
3. 每张 Agent 卡完成/失败
4. 任务完成
5. 任务失败
6. 任务进入人工审核

`progress.publish` 内部：
```python
async def publish(event_type, task_id, execution_id, payload):
    if not settings.progress_publish_enabled:
        return  # 阶段 2 no-op；阶段 3 打开配置后生效
    await channel.default_exchange.publish_to(...)
```

阶段 3 打开后，progress payload 只承诺服务端内部字段，不承诺前端展示字段。Java subscriber 收到事件后负责转换为当前 SSE 合约；如果某个 Agent 完成事件只带 `agentName/runAttempt`，Java 必须查 `agent_run_log` 补齐 `thinking/evidence/suggestion/reasonWhy` 后再发给浏览器。

### 6.6 AES-GCM 密钥解密（新增）

当前 Python 的 `crypto_utils.py` 只有 Fernet 实现，和 Java `AesGcmUtil`（AES-256-GCM，`IV(12) || ciphertext || tag(16)`，Base64）**完全不兼容**。HTTP 派发链路下 Java 已经解密好把明文放进请求体发过来，所以一直没暴露问题。改 MQ 后 Java 不再把明文放进请求体，而是在创建任务时把 `user_llm_config` 的加密字段复制到 `pricing_task.llm_api_key_enc`；Python 从 `pricing_task` 读密文，必须能用同一把 key 解出 Java 写入的密文。

在 `crypto_utils.py` 里新增：

```python
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
import base64

IV_LENGTH = 12
TAG_LENGTH = 16

def _aes_key_bytes() -> bytes:
    """Java 用 Base64-encoded 32-byte key；Python 原样解码即可。"""
    secret = get_settings().llm_key_encryption_secret
    return base64.b64decode(secret)

def decrypt_api_key_gcm(cipher_b64: str) -> str:
    """兼容 Java AesGcmUtil.encrypt 的 AES-256-GCM 密文。

    Layout: IV(12) || ciphertext || tag(16)，整体 Base64。
    """
    raw = base64.b64decode(cipher_b64)
    if len(raw) < IV_LENGTH + TAG_LENGTH:
        raise ValueError("ciphertext too short for AES-GCM")
    iv = raw[:IV_LENGTH]
    ct_with_tag = raw[IV_LENGTH:]
    aes = AESGCM(_aes_key_bytes())
    return aes.decrypt(iv, ct_with_tag, None).decode("utf-8")
```

Python 侧保留 `encrypt_api_key / decrypt_api_key`（Fernet）给历史 `pricing_task.llm_api_key_enc` 和开发态数据使用。新业务路径推荐提供统一入口：

```python
def decrypt_task_api_key(ciphertext: str) -> str:
    try:
        return decrypt_api_key_gcm(ciphertext)
    except Exception:
        return decrypt_api_key(ciphertext)
```

这样 Java 新复制的 AES-GCM 密文可以直接解密，旧任务中已经存在的 Fernet 密文也不会在过渡期失效。等历史任务全部完成并确认不再依赖 Fernet 后，再移除 fallback。

`LLM_KEY_ENCRYPTION_SECRET` 环境变量 Java/Python 共享（docker-compose 注入同一个值）。

### 6.7 启动期 RabbitMQ 不可达的处理

`rabbitmq_worker_service.start()` 自己维护重试循环，循环内使用 `aio_pika.connect_robust(...)`；连接失败只记录 WARN 并 sleep/backoff 后重试，不阻塞 FastAPI 启动：

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    task = asyncio.create_task(rabbitmq_worker_service.start())
    try:
        yield
    finally:
        task.cancel()
        await rabbitmq_worker_service.stop()
```

`rabbitmq_worker_service.start()` 内部：
- 首次连接失败：记录 WARN，设 `self._ready = False`，等待 backoff 后重新调用 `connect_robust`。
- 连接成功后：声明 dispatch 拓扑、开始 consume，设 `self._ready = True`；后续断线由 robust connection 处理，恢复后重新声明拓扑并继续 consume。
- Python 的 `/health/ready` 继续检查 DB，并追加 RabbitMQ worker ready 状态：只有 `dbOk && rabbitmqOk` 时返回 `status=ok`；响应体暴露 `{"rabbitmq": "ok" if worker.ready else "down"}`。

### 7.3 RabbitMqConfig（Jackson 转换器是强制项）

**这是整个方案最容易踩坑的点**：不注册 `Jackson2JsonMessageConverter`，Spring AMQP 默认会用 `SimpleMessageConverter` 做 JDK 原生序列化，Python `json.loads` 直接失败。

```java
@Configuration
public class RabbitMqConfig {

    @Value("${app.rabbitmq.dispatch-exchange}") private String dispatchExchange;
    @Value("${app.rabbitmq.dispatch-queue}") private String dispatchQueue;
    @Value("${app.rabbitmq.dispatch-routing-key}") private String dispatchRoutingKey;

    @Bean
    public DirectExchange dispatchExchange() {
        return ExchangeBuilder.directExchange(dispatchExchange).durable(true).build();
    }

    @Bean
    public Queue dispatchQueue() {
        return QueueBuilder.durable(dispatchQueue).build();
    }

    @Bean
    public Binding dispatchBinding(Queue dispatchQueue, DirectExchange dispatchExchange) {
        return BindingBuilder.bind(dispatchQueue).to(dispatchExchange).with(dispatchRoutingKey);
    }

    @Bean
    public MessageConverter jsonMessageConverter(ObjectMapper objectMapper) {
        // 复用 Spring Boot 托管的 ObjectMapper（带 JavaTimeModule，
        // 可直接序列化 Instant 为 ISO-8601 字符串）
        return new Jackson2JsonMessageConverter(objectMapper);
    }

    @Bean
    public RabbitTemplate rabbitTemplate(ConnectionFactory cf, MessageConverter jsonMessageConverter) {
        RabbitTemplate template = new RabbitTemplate(cf);
        template.setMessageConverter(jsonMessageConverter);
        template.setMandatory(true);
        template.setReturnsCallback(returned -> {
            // 这里不要吞掉 return。实际实现可记录到 TaskDispatchPublisher
            // 的失败引用里，再由 publishAndConfirm 统一抛异常。
            throw new IllegalStateException("dispatch message returned: " + returned.getReplyText());
        });
        // PERSISTENT 在 application.yml 里通过 spring.rabbitmq.template.delivery-mode 配好
        return template;
    }

    @Bean
    public SimpleRabbitListenerContainerFactory rabbitListenerContainerFactory(
            ConnectionFactory cf, MessageConverter jsonMessageConverter) {
        SimpleRabbitListenerContainerFactory factory = new SimpleRabbitListenerContainerFactory();
        factory.setConnectionFactory(cf);
        factory.setMessageConverter(jsonMessageConverter);
        factory.setAcknowledgeMode(AcknowledgeMode.MANUAL);
        factory.setPrefetchCount(1);
        return factory;
    }
}
```

`ObjectMapper` 里要有 `JavaTimeModule` 并启用 `WRITE_DATES_AS_TIMESTAMPS=false`（Spring Boot 默认就是）——这样 `Instant` 自动序列化为 `"2026-04-21T04:39:00.123Z"`，Python 直接 `datetime.fromisoformat` 能解。

实际发送不要在业务 service 里裸调 `rabbitTemplate.convertAndSend`。新增一个小的 `TaskDispatchPublisher` 包装：

1. 生成 `CorrelationData(eventId)`。
2. 调用 `rabbitTemplate.convertAndSend(exchange, routingKey, event, correlationData)`。
3. 等待 `correlationData.getFuture()` 在短超时内返回。
4. confirm ack 才返回成功；confirm nack、future 超时、mandatory return 都抛出 `AmqpException` 或 `IllegalStateException`。

`startTask` 第 2 步只依赖这个 publisher 的成功/失败，不直接关心 RabbitTemplate 细节。

### 7.4 startTask 改造（三步独立小事务 + 非事务外壳）

原来：

```java
task.setTaskStatus("PENDING");
taskMapper.insert(task);
try {
    dispatchToPython(task.getId(), ...);
} catch (Exception e) {
    task.setTaskStatus("FAILED"); ...
    throw ...;
}
```

改为：

```java
// ⚠️ 注意：startTask 本身不再加 @Transactional！
//         如果加了，rabbitTemplate.convertAndSend 会在 DB commit 之前把消息发到 broker，
//         Python 消费者可能查不到 pricing_task 行（幽灵消息 → 任务丢失）。
public Long startTask(...) {
    // ... 前面的校验 / 查 LLM 配置 / 计算 idempotencyKey / findReusableTask ...
    // Java 不解密 LLM Key，只把密文字段复制到 pricing_task，Python 用 AES-GCM 解密。
    task.setLlmApiKeyEnc(llmConfig.getLlmApiKeyEnc());
    task.setLlmBaseUrl(llmConfig.getLlmBaseUrl());
    task.setLlmModel(llmConfig.getLlmModel());

    if (existingTask != null) {
        // 复用命中，不重复 publish。Python 若正在处理会自然完成；
        // 已完成 / 人工审核态直接把旧结果给用户；Java 什么都不用做。
        return existingTask.getId();
    }

    // 第 1 步：独立事务插入 PENDING 并提交
    Long taskId = transactionTemplate.execute(status -> {
        task.setTaskStatus("PENDING");
        taskMapper.insert(task);
        return task.getId();
    });

    // 第 2 步：publish + broker confirm（此时 DB 已 commit，Python 一定能查到）
    TaskDispatchEvent event = new TaskDispatchEvent(
        UUID.randomUUID().toString(),
        taskId,
        task.getTraceId(),
        Instant.now()
    );
    try {
        taskDispatchPublisher.publishAndConfirm(event);
    } catch (RuntimeException e) {
        log.error("publish dispatch failed, taskId={}", taskId, e);
        transactionTemplate.executeWithoutResult(status ->
            taskMapper.updateStatusAndReason(taskId, "FAILED", truncate("派发失败: " + e.getMessage(), 255))
        );
        throw new IllegalStateException("任务派发失败: " + e.getMessage(), e);
    }

    // 第 3 步：UPDATE QUEUED。带 WHERE task_status='PENDING' 守卫，
    //          避免 Python 已经 CAS 成 RUNNING 时被回写 QUEUED。
    //          即便此步失败，§15 孤儿扫描也会补发 PENDING 并转 QUEUED；
    //          重复消息由 execution fencing 消化。
    transactionTemplate.executeWithoutResult(status ->
        taskMapper.updateStatusIfPending(taskId, "QUEUED")
    );
    return taskId;
}
```

这里刻意用 `TransactionTemplate`，不写同类内 `@Transactional` 私有/公有方法：Spring AOP 自调用不会经过事务代理，按同类方法拆事务会失效。

SQL 层：
```sql
-- updateStatusIfPending
UPDATE pricing_task SET task_status = #{status}, updated_at = NOW()
 WHERE id = #{taskId} AND task_status = 'PENDING';
```

三段式状态的意义：
- `PENDING` 表示"Java 已插入行，消息是否已入 broker 未知"——短暂窗口，靠 §15 兜底。
- `QUEUED` 表示"消息已入 broker，等 Worker 认领"。
- `RUNNING` 表示"Worker 已 CAS 占用并开始跑"。

### 7.5 HealthController 字段调整

项目约束要求 Java `/api/health/ready` 继续依赖 Python `/health/ready`，不能因为派发链路改 MQ 就删除 Python readiness。这里的改造是**保留 `pythonWorker`，追加 `rabbitmq` 字段**：只有 DB、Python、RabbitMQ 都 ready 时 Java ready 才是 `ok`。

```java
@GetMapping("/ready")
public Map<String, Object> ready() {
    Map<String, Object> payload = new LinkedHashMap<>();
    boolean databaseOk = false;
    boolean pythonOk = pythonBackendHealthClient.isReady();
    try {
        jdbcTemplate.queryForObject("SELECT 1", Integer.class);
        databaseOk = true;
    } catch (Exception ignore) {}

    // 用 Spring AMQP 的 ConnectionFactory 自带探测；
    // createConnection() 在连上的情况下会复用底层 Connection，无副作用。
    boolean rabbitmqOk = false;
    try (Connection ignored = connectionFactory.createConnection()) {
        rabbitmqOk = true;
    } catch (Exception ignore) {}

    payload.put("status", databaseOk && pythonOk && rabbitmqOk ? "ok" : "degraded");
    payload.put("database", databaseOk ? "ok" : "down");
    payload.put("pythonWorker", pythonOk ? "ok" : "down");
    payload.put("rabbitmq", rabbitmqOk ? "ok" : "down");
    return payload;
}
```

字段 `pythonWorker` **不移除**，避免破坏现有 readiness 语义和运维脚本；新增 `rabbitmq` 供 MQ 链路观测。

## 8. 批量定价链路的适配

`PricingBatchServiceImpl.createBatch` 循环里每次调 `decisionTaskService.createPricingTask(productId, ...)` 会自动走新 publish 路径，行为不变、**无需改这个类的主逻辑**。50 个商品 = 50 条独立消息。

`recoverTaskAfterCreateFailure` 原本是 HTTP 派发失败时查历史 task 复用的逻辑。改 MQ 后，派发失败只剩"publish 到 RabbitMQ 失败"一种情况，此时 `startTask` 会在第 2 步 catch 中把 task 置为 `FAILED` 后再抛异常。再调 `recoverTaskAfterCreateFailure` 可能会"找到刚刚 FAILED 的 task 并错误复用"，风险比之前还大——所以**直接删除这段恢复代码**，让 `PricingBatchItem` 标记 `CREATE_FAILED` 即可。同时也一并消除了前一轮 review 提出的"误关联历史 FAILED 任务"风险。

```java
private final Map<Long, List<SseEmitter>> emitters = new ConcurrentHashMap<>();

public void register(Long taskId, SseEmitter emitter) {
    emitters.computeIfAbsent(taskId, k -> new CopyOnWriteArrayList<>()).add(emitter);
    emitter.onCompletion(() -> unregister(taskId, emitter));
    emitter.onTimeout(() -> unregister(taskId, emitter));
}

@RabbitListener(queues = "${app.rabbitmq.progress-queue}")
public void onProgress(TaskProgressEvent event, Channel channel,
                       @Header(AmqpHeaders.DELIVERY_TAG) long tag) throws IOException {
    try {
        if (!isCurrentExecution(event.getTaskId(), event.getExecutionId())) {
            channel.basicAck(tag, false);
            return;
        }

        List<Map<String, Object>> payloads = ssePayloadFactory.fromProgressEvent(event);
        for (SseEmitter em : emitters.getOrDefault(event.getTaskId(), List.of())) {
            for (Map<String, Object> payload : payloads) {
                try {
                    em.send(SseEmitter.event().name("message").data(payload));
                } catch (Exception sendError) {
                    unregister(event.getTaskId(), em);
                }
            }
        }
        channel.basicAck(tag, false);
    } catch (Exception ex) {
        // progress 是实时通知，不是真相源；失败时 ack 避免卡死队列，
        // 前端重连或手动刷新会通过 DB 快照补齐。
        channel.basicAck(tag, false);
    }
}
```

`stream(taskId, userId)` 改为：

## 14. 总结

本方案相比"正式上线版"删除了 Outbox、多级重试、DLX、灰度开关、HA 等所有生产级设施，但**保留并加强了业务正确性的核心**：

- **事务边界与消息边界彻底分离**：`TransactionTemplate` 三段独立小事务 + 外层非事务，避免 publish-before-commit 的幽灵消息
- **publisher confirm 收口派发成功语义**：只有 broker confirm ack 且没有 mandatory return 才把任务置为 `QUEUED`，避免"假成功入队"
- **execution fencing 覆盖全部写回路径**：普通 CAS 以 `IS NULL` 抢占，redelivered 消息可接管旧 execution；UPDATE 带 `current_execution_id = ?` 和终态守卫，INSERT/UPSERT 都带 owner guard
- **取消后不被旧 attempt 覆盖**：Java 取消时清空 `current_execution_id`，Python 所有写回排除 `CANCELLED`
- **消息体最小化 + 双端对齐的 AES-GCM**：消息只带 `taskId`；Java 把用户 LLM 密文字段复制到 `pricing_task`，Python 新增 `decrypt_task_api_key`，AES-GCM 优先并兼容旧 Fernet 密文
- **独立的 `consumer_retry_count`**：不和业务层重试共用额度
- **Jackson JSON 序列化 + ISO-8601 UTC 时间**：Java 和 Python 双方能直接互读
- **PROGRESS_PUBLISH_ENABLED 开关 + SSE 合约不变**：阶段 2 独立上线时 progress 链路完全沉默；阶段 3 打开后 Java 仍向前端发送现有 `task_started/agent_card/task_completed/task_failed` payload
- **HealthController 保留 Python readiness 并追加 RabbitMQ readiness**：不破坏既有 `/api/health/ready` 语义
- **PENDING 补发扫描**：覆盖极罕见的"insert 成功但 publish 未完成"窗口，且不会误杀已经 publish 但未转 QUEUED 的任务

能解决三个真实痛点（Worker 恢复、晚写覆盖、SSE 不再轮询 DB），并能显著缓解"数据库兼职任务队列"带来的扩展性和调度耦合问题。保留原设计里 MySQL 真相源的分层原则，代价为 **3-4.5 天实施 + 1 份 migration（加 4 列 + 2 个索引）+ 不引入新业务表**。

阶段 2 是核心必做（解决"Worker 恢复 + 晚写覆盖"），阶段 3 SSE 去轮询是可选优化。后续如果要上线再按正式版补 Outbox、DLX、HA 即可，本次改动不会成为后续升级的障碍。

## 15. 遗留窗口与兜底：PENDING 补发扫描

三段式 startTask 有两个极罕见窗口：
- **第 1 步 insert 完成（已 commit）→ 第 2 步 publish 前 Java 进程被强杀**：DB 有 `PENDING`，broker 没消息。
- **第 2 步 publish 成功 → 第 3 步 UPDATE QUEUED 前 Java 进程被强杀**：DB 仍是 `PENDING`，broker 已有消息。

因此兜底不能直接把旧 `PENDING` 标记为 `FAILED`，否则第二种窗口会误杀可执行任务。正确做法是：对 5 分钟前仍为 `PENDING` 的任务补发一条 dispatch 消息，然后用 `WHERE task_status='PENDING'` 转 `QUEUED`。重复消息由 execution fencing 消化。

`StartupReconcileService` 在 `ApplicationReadyEvent` 里跑一次，之后每 5 分钟跑一次。注意：项目启动类需要加 `@EnableScheduling`。

```java
@Component
@RequiredArgsConstructor
public class StartupReconcileService {
    private final PricingTaskMapper taskMapper;
    private final TaskDispatchPublisher taskDispatchPublisher;

    @EventListener(ApplicationReadyEvent.class)
    @Scheduled(fixedDelay = 5 * 60 * 1000)
    public void reconcileOrphanPendingTasks() {
        // 只扫 5 分钟前的 PENDING。刚创建的 PENDING 可能正处在第 2/3 步之间，不动。
        for (PricingTask task : taskMapper.selectStalePendingTasks(5)) {
            TaskDispatchEvent event = new TaskDispatchEvent(
                UUID.randomUUID().toString(),
                task.getId(),
                task.getTraceId(),
                Instant.now()
            );
            try {
                taskDispatchPublisher.publishAndConfirm(event);
                taskMapper.updateStatusIfPending(task.getId(), "QUEUED");
                log.warn("re-published stale PENDING task {}", task.getId());
            } catch (Exception ex) {
                // 保持 PENDING，下一轮继续补发；不要误标 FAILED。
                log.warn("re-publish stale PENDING task failed, taskId={}", task.getId(), ex);
            }
        }
    }
}
```

当前仓库 `PricingTaskMapper` 是 MyBatis-Plus `BaseMapper` 接口，没有既有 XML mapper。这里优先用注解方法，避免为了两条 SQL 引入新的 XML 风格：

```java
@Select("""
    SELECT *
      FROM pricing_task
     WHERE task_status = 'PENDING'
       AND created_at < NOW() - INTERVAL #{minutes} MINUTE
    """)
List<PricingTask> selectStalePendingTasks(@Param("minutes") int minutes);

@Update("""
    UPDATE pricing_task
       SET task_status = #{status},
           updated_at = NOW()
     WHERE id = #{taskId}
       AND task_status = 'PENDING'
    """)
int updateStatusIfPending(@Param("taskId") Long taskId, @Param("status") String status);
```
