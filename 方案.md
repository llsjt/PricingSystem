# 批量定价 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在不改变当前 Python 单商品执行语义、不破坏现有单任务 SSE / 决策档案 / 应用建议链路的前提下，为商品列表多选提供批量定价编排、批量进度页和逐条应用能力。

**Architecture:** 批量定价只新增 Java 批次编排层和前端批次进度视图。每个商品仍然对应一条现有 `pricing_task`；新表 `pricing_batch` / `pricing_batch_item` 只负责把多条单任务组织成一个批次。批次页通过 Java 轮询聚合 `pricing_batch_item + pricing_task + pricing_result`，并复用现有 `/archive` 详情和 `/api/decision/apply/{resultId}`。

**Tech Stack:** Vue 3 + TypeScript + Element Plus + Vue Router；Spring Boot 3.2 + MyBatis-Plus；MySQL；FastAPI/Python worker 继续沿用现有队列和 4-Agent 流程。

## 0. 当前真实契约（执行时以现有代码为准）

以下约束来自当前仓库实现，本方案全部以这些真实行为为基线，不再沿用旧版错误假设：

- 单商品创建入口是 `DecisionTaskService.createPricingTask(productId, strategyGoal, constraints, userId)`；它内部仍走 `startTask(List<Long> productIds, ...)`，并且当前只取 `productIds.get(0)`。因此 `POST /api/decision/start` 不能作为真实批量入口。
- `pricing_task.task_status` 当前真实生命周期是：
  `PENDING -> QUEUED -> RUNNING -> RETRYING -> RUNNING -> ... -> MANUAL_REVIEW / COMPLETED / FAILED / CANCELLED`
- `MANUAL_REVIEW` 当前仍然是 `pricing_task.task_status` 的真实终态之一。前后端、SSE、运营指标都在按这个状态工作，不能把它当成“只存在于 `pricing_result.execute_strategy`”。
- `pricing_result.execute_strategy` 当前真实值是中文 `人工审核`；它是结果元数据，不用于区分任务是否处于“待人工审核”阶段。
- 当前 Python 结果写入会把 `pricing_result.review_required = 1`，并把任务状态写成 `MANUAL_REVIEW`；当用户执行 `/api/decision/apply/{resultId}` 后，Java 会把 `review_required` 改成 `0`，并把 `task_status` 改成 `COMPLETED`。
- `createPricingTask()` 在 Python 拒绝受理（例如队列满）时，**可能已经创建出一条 `pricing_task` 并被标记为 `FAILED`**，然后 Java 再抛异常。不能再假设“抛异常就一定没有 `task_id`”。
- 前端左侧导航高亮当前是 `route.path.startsWith(nav.path)`；因此 `/archive/batches/:batchId` 天然会高亮“决策档案”，不需要为了高亮强行嵌套路由或新增 `activePath`。
- 但 `BasicLayout.vue` 的页面标题和页签标题当前优先取导航标题；如果新增 `/archive/batches/:batchId`，想让标题显示“批量定价进度”，需要顺手调整标题来源。
- `product` 表真实列名是 `product_name` / `sale_price`；Java 实体映射为 `Product.title` / `Product.currentPrice`。文档和实现都不能再写成 `title` / `current_price` 这种不存在的 SQL 列。
- 仓库当前没有 MyBatis XML mapper，也没有复杂 `@Select` join 查询的既有模式。本期批次查询应优先走 MyBatis-Plus 基础查询 + 服务层组装。

## 2. 修订后的总体方案

采用“批次父记录 + 批次明细 + 复用现有单商品任务”的方式。

用户在 `ProductList.vue` 中多选商品，统一设置策略目标和约束。Java 创建一条 `pricing_batch` 后，按商品逐条调用现有 `DecisionTaskService.createPricingTask(...)`。每个商品最终会落成以下两种批次明细之一：

- `TASK_LINKED`：成功拿到一条 `pricing_task`，不论这条任务当前是 `PENDING`、`RUNNING`、`MANUAL_REVIEW`、`COMPLETED` 还是 `FAILED`。
- `CREATE_FAILED`：批次层最终没有拿到任何 `task_id`，例如商品不存在、权限不足、系统级校验失败、或异常发生在任务落库之前。

关键修订点如下：

1. 批次统计不再用 `pricing_result.execute_strategy` 区分“待人工审核”，而是直接用 `pricing_task.task_status = MANUAL_REVIEW`。
2. 批次明细不再把 `item_status` 冒充任务运行态，而是同时返回：
   - `creationStatus`
   - `taskStatus`
   - `displayStatus`
3. 队列满导致的“任务已落库但派发失败”不再被误记成 `CREATE_FAILED`；批次服务要尽量把这条 `FAILED` 任务找回来并关联到明细。
4. 批次详情查询不写复杂 XML/注解 SQL；由于单批最多 50 条，服务层直接把 `pricing_batch_item` 全量读出后批量加载 `product` / `pricing_task` / `pricing_result` 组装即可。

### 3.1 状态定义

批次 `batch_status`：

- `RUNNING`：存在 `PENDING / QUEUED / RUNNING / RETRYING` 的子任务。
- `MANUAL_REVIEW`：全部已到终态，且至少一条子任务状态为 `MANUAL_REVIEW`，同时没有失败或取消。
- `COMPLETED`：全部已到终态，且全部子任务状态为 `COMPLETED`。
- `PARTIAL_FAILED`：没有运行中的任务，且存在失败或取消，同时至少存在一条非失败项。
- `FAILED`：全部都失败类结束（`FAILED` 或 `CREATE_FAILED`）。
- `CANCELLED`：全部都被取消结束。

明细 `item_status`：

- `TASK_LINKED`：批次层拿到了 `task_id`。这个任务后续可能进入任意真实 `task_status`。
- `CREATE_FAILED`：批次层最终没有拿到 `task_id`。

明细展示状态 `displayStatus` **不入库**，仅在接口层派生：

```text
if item_status == CREATE_FAILED -> CREATE_FAILED
else                           -> pricing_task.task_status
```

## 4. 聚合与状态派生规则

由于一期单批上限就是 50，聚合与分页都在服务层基于全量 50 条以内明细做内存组装：

- 先按 `batch_id` 读取全部 `pricing_batch_item`
- 再批量读取关联 `product`
- 再批量读取关联 `pricing_task`
- 再批量读取关联 `pricing_result`
- 最后在 Java 内存里派生计数、排序、过滤、分页和 VO

计数规则严格按当前真实契约：

```text
runningCount      = count(taskStatus in [PENDING, QUEUED, RUNNING, RETRYING])
completedCount    = count(taskStatus = COMPLETED)
manualReviewCount = count(taskStatus = MANUAL_REVIEW)
failedCount       = count(taskStatus = FAILED) + count(itemStatus = CREATE_FAILED)
cancelledCount    = count(taskStatus = CANCELLED)
totalCount        = count(pricing_batch_item)
```

注意：

- `MANUAL_REVIEW` 计数来自 `pricing_task.task_status`，不是 `pricing_result.execute_strategy`。
- `execute_strategy` 当前真实值基本固定为 `人工审核`，可返回给前端展示，但不能拿它推导批次状态。
- `review_required` 用于行级展示，不用于批次聚合。

批次状态派生规则：

```text
if runningCount > 0                         -> RUNNING
else if totalCount == cancelledCount        -> CANCELLED
else if totalCount == failedCount           -> FAILED
else if failedCount + cancelledCount > 0    -> PARTIAL_FAILED
else if manualReviewCount > 0               -> MANUAL_REVIEW
else                                        -> COMPLETED
```

批次详情默认只做实时聚合；只在以下时机把聚合结果回写到 `pricing_batch`：

- `createBatch()` 循环结束后
- `cancelBatch()` 完成后
- 详情轮询时首次发现批次进入终态，且 `finalized_at IS NULL`

## 7. 关键实现细节

### 7.1 批量创建循环

`createBatch()` 的推荐流程：

1. 预校验整批请求。
2. 插入 `pricing_batch`。
3. 按去重后的 `productIds` 顺序逐条处理，每条独立提交。
4. 每条商品调用现有 `decisionTaskService.createPricingTask(...)`。
5. 成功拿到 `taskId` 时写入一条 `TASK_LINKED` 明细。
6. 如果抛异常，先尝试按“同一幂等键”找回刚刚创建但派发失败的任务：
   - 找到任务：写入 `TASK_LINKED`，并带上这个 `taskId`
   - 找不到任务：写入 `CREATE_FAILED`
7. 循环结束后聚合一次批次统计并回写 `pricing_batch`。

### 7.2 幂等键与失败恢复

当前 `buildIdempotencyKey()` 和 `findReusableTask()` 都在 `DecisionTaskServiceImpl` 私有方法里。为了让批次服务能在异常时“找回已落库但失败的任务”，需要把这部分能力抽成共享支持类：

- `PricingTaskReuseSupport.buildIdempotencyKey(userId, productIds, strategyGoal, constraints)`
- `PricingTaskReuseSupport.findReusableTask(idempotencyKey, shopId)`
- `PricingTaskReuseSupport.findLatestTaskByIdempotencyKey(idempotencyKey, shopId)`

约束：

- `DecisionTaskServiceImpl` 继续使用“可复用状态 = `QUEUED / RUNNING / RETRYING / MANUAL_REVIEW / COMPLETED`”。
- 批次服务的“失败恢复查询”不能只看可复用状态；它必须能找回刚被标记成 `FAILED` 的任务。

### 7.3 查询与分页

由于单批上限是 50，一期不写自定义 join SQL：

- `getBatchDetail(batchId)`：读取整批 items 后在内存里聚合。
- `getBatchItems(batchId, page, size, status)`：读取整批 items，组装 `displayStatus` 后在内存里过滤和分页。

这比在当前仓库里临时引入 XML mapper 或注解 SQL 更符合已有实现风格，也足够支撑一期规模。

### 7.4 查看详情与应用建议

批次明细页的两个操作完全复用现有链路：

- `查看详情`：跳转到 `/archive?taskId=<taskId>`，复用当前 `Archive.vue` 打开任务详情抽屉的行为。
- `应用建议`：直接调用 `applyDecision(resultId)`。

按钮规则：

- `resultId` 为空：不显示按钮。
- `appliedStatus = 已应用`：显示“已应用”禁用态。
- `displayStatus = MANUAL_REVIEW`：按钮文案显示“人工审核并应用”。
- `displayStatus = COMPLETED` 且仍未应用：按钮文案显示“应用建议”。
- `displayStatus = CREATE_FAILED`：只显示错误信息，不显示操作按钮。

## 8. 与现有链路的兼容性

本方案不会破坏现有单商品链路，原因如下：

- `pricing_task` 仍然保持“一条任务只对应一个商品”。
- `pricing_result` 仍然保持“一条结果只对应一个任务”。
- Python worker 仍然只消费单个 `task_id + product_id`。
- 单商品智能定价页 `/lab`、单任务 SSE、任务抽屉、日志、结果报告都保持不变。
- `DecisionController.applyDecision()` 继续作为唯一价格应用入口。
- 批次详情只是聚合视图，不引入第二套结果或应用逻辑。

## 9. 风险与控制

### 9.1 队列满 / 派发拒绝

当前真实行为是：

- Java 先创建 `PENDING` 任务
- Python 可能拒绝并把任务标成 `FAILED`
- Java 再抛异常

一期不试图“伪装成没有任务”；而是明确把这条 `FAILED` 任务恢复并挂到批次明细上。这样用户至少能在批次页看到真实失败任务，而不是看到一条莫名其妙的 `CREATE_FAILED`。

### 9.2 事务边界

- 不要用一个大事务包住整个批次循环。
- 推荐“批次主记录先插入 + 每个明细单独提交 + 最后再更新批次快照”的模式。
- 单个商品失败不能回滚前面已经成功写入的明细。

### 9.3 幂等

- 单商品幂等仍沿用当前 `pricing_task.idempotency_key`。
- 批次本身一期不做请求级幂等。
- 用户连续点击两次“批量定价”会产生两个 `pricing_batch`，但其中部分商品可能复用同一条已有 `pricing_task`；这是当前真实行为，文档和前端提示都要接受它。

### 9.4 权限

- 创建批次前校验全部商品归属。
- 查询详情 / 明细 / 取消批次时校验 `pricing_batch.requested_by_user_id`。
- 应用价格时继续依赖现有 `applyDecision(resultId)` 的权限校验。

## 12. 回滚策略

如果一期上线后出现问题：

1. 前端先隐藏商品列表中的“批量定价”入口。
2. Java 保留批次接口但不再提供前端入口。
3. 已经创建的 `pricing_task` 继续按现有单任务逻辑执行，不需要回滚单任务表。
4. `pricing_batch` / `pricing_batch_item` 可保留作为历史数据，不影响现有业务主链路。
5. 如必须回滚数据库，先确认没有依赖批次表的线上流程，再删除批次两张表。

## 13. 最终结论

修订后的方案仍然采用“批次编排层 + 复用现有单任务”的总体思路，但所有统计、状态、失败恢复、路由和字段来源都改成了**基于当前真实契约**的版本：

- `MANUAL_REVIEW` 继续视为真实 `task_status`
- `execute_strategy` 只作为结果元数据返回
- 队列满场景按“可能已存在 FAILED 任务”处理
- 批次明细显式区分创建状态、任务状态和展示状态
- `/archive/batches/:batchId` 走扁平路由，配合 `BasicLayout` 标题修正
- 一期查询全部采用 MyBatis-Plus + 服务层组装，不引入和当前仓库风格不一致的复杂 SQL 基础设施

按这版文档实施，能够在不动 Python 批量语义、不重写单任务链路的前提下，得到一条可交付、可回滚、与现有代码一致的批量定价实现路径。
