# 商品操作栏跳转智能定价完整方案

## 目标

在“商品数据管理”的表格操作栏中新增“定价”按钮。用户点击后跳转到“智能定价”页面，并尽可能自动带入当前商品的定价上下文（平台、店铺、商品），减少重复选择的操作。

该功能只负责页面跳转与表单预填，不自动创建定价任务，不改变现有智能定价任务创建、SSE 推送、决策档案和价格应用逻辑。

为了避开智能定价页现有商品下拉 `loadProducts(keyword)` "只支持名称关键词搜索、不支持按 id 精确查询" 的签名限制，跳转 URL 会额外携带 `productName`，智能定价页用它作为关键词调远程搜索，再按 `id` 精确匹配目标商品（即下文所述的"方案 C"）。

## 推荐方案

采用“商品列表传 query，智能定价页读取 query 并预填表单”的方案。

跳转地址格式：

```text
/lab?productId=123&shopId=1&platform=天猫&productName=某商品名称
```

字段含义：

- `productId`：商品 ID，用于在搜索结果中精确定位并预选商品。
- `shopId`：商品所属店铺 ID，用于准确预选店铺。
- `platform`：商品所属平台，用于预选平台。
- `productName`：商品名称，用于作为关键词调用智能定价页现有的 `loadProducts(keyword)` 远程搜索接口，确保目标商品出现在下拉候选集内。

智能定价页收到参数后只做预填：

- 成功匹配时：平台、店铺、商品自动选中。
- 无法匹配时：保留页面可用状态，提示用户手动选择。
- 不自动启动任务：用户仍需点击“启动任务”。

## 为什么需要返回 shopId

只传 `productId + platform` 在单店铺场景下可用，但在同一平台多店铺场景下会有歧义。

例如：

- 店铺 A：天猫
- 店铺 B：天猫
- 商品 X 属于店铺 B

如果只传 `platform=天猫`，智能定价页无法知道应该选择店铺 A 还是店铺 B。当前智能定价页的商品下拉会按 `shopId` 查询商品，所以缺少 `shopId` 会导致预填不稳定。

因此建议后端商品列表接口额外返回 `shopId`。这是向后兼容新增字段，不影响现有前端页面。

## 为什么需要带 productName

智能定价页现有的商品下拉是远程搜索模式，内部实现为：

```ts
const loadProducts = async (keyword = '') => {
  ...
  const res = await getProductList({ page: 1, size: 100, keyword, platform: ..., shopId: ... })
  productOptions.value = Array.isArray(res.data?.records)
    ? res.data.records.map((item) => ({ id: Number(item.id), productName: String(item.productName || '') }))
    : []
}
```

`loadProducts` 的唯一参数是**商品名称关键词**，最终会作为 `keyword` 透传给 `/products/list` 的模糊搜索。它不支持按 `id` 精确查询。

这意味着：

- 如果跳转时只传 `productId`，智能定价页要预填商品下拉，必须先触发一次 `loadProducts`，但没有合适的关键词可用。
- 如果硬把 `productId` 当作 keyword 传入（例如 `loadProducts(String(productId))`），后端会按名称去模糊匹配数字字符串，**除非商品名里恰好包含该 ID 数字，否则搜不到目标商品**，导致预填失败并回落到"未找到商品"分支。
- 如果不传 keyword（`loadProducts('')`），后端返回的是店铺下前 100 条商品——当店铺 SKU 超过 100 条时目标商品可能不在候选集内，仍会预填失败。

最可靠的做法是：商品列表跳转时**顺带把当前行的 `productName` 塞进 URL**，智能定价页用它作为关键词调 `loadProducts`，然后在返回结果里按 `id` 精确匹配。这样：

- 不需要新增后端接口或修改 `/products/list` 查询参数。
- 不依赖"店铺下商品数 ≤ 100"这种隐式假设。
- 即使商品名在源端与目的端存在轻微差异（例如尾部空格），也能通过命中候选集后的 id 精确比对兜底。

如果仍然没匹配到（极少数情况：名称字段完全不同、商品被另一账号下架等），再回落到 `loadProducts('')` 拉默认列表再尝试，最后兜底才提示用户手动搜索。

### 3. 增加根据路由预填方法

文件：

- `frontend/src/views/PricingLab.vue`

新增：

```ts
const prefillFromRoute = async () => {
  const productId = parsePositiveId(route.query.productId)
  const shopId = parsePositiveId(route.query.shopId)
  const platform = String(route.query.platform || '').trim()
  const productName = String(route.query.productName || '').trim()

  // 必要参数：productId。缺它则完全无法预填商品，直接退出走默认初始化流程。
  if (!productId) return false

  // ── 步骤 1：确定平台 ─────────────────────────────
  // 优先使用 URL 显式传入的 platform；缺失时尝试用 shopId 反查店铺拿到 platform。
  if (platform) {
    taskConfig.platform = platform
  } else if (shopId) {
    const matchedShop = shopStore.shops.find((shop) => Number(shop.id) === shopId)
    taskConfig.platform = matchedShop?.platform || ''
  }

  if (!taskConfig.platform) {
    ElMessage.warning('未找到商品所属平台，请手动选择平台后继续定价')
    activeStep.value = 0
    return true
  }

  productOptions.value = []

  // ── 步骤 2：确定店铺 ─────────────────────────────
  // URL 明确带了 shopId 且当前用户有权访问 → 直接用；否则单店铺自动选中，多店铺提示手选。
  if (shopId && availableShops.value.some((shop) => Number(shop.id) === shopId)) {
    taskConfig.shopId = shopId
  } else if (availableShops.value.length === 1) {
    taskConfig.shopId = availableShops.value[0].id
  } else {
    ElMessage.info('请选择商品所属店铺后继续定价')
    activeStep.value = 0
    return true
  }

  // ── 步骤 3：加载商品候选并按 id 精确匹配 ───────────
  // loadProducts 签名仅支持 keyword 关键词模糊搜索，不支持按 id 精确查询。
  // 方案 C：用商品列表传过来的 productName 作为关键词，再在返回结果里按 id 精确匹配，
  // 这样既复用现有接口又能稳定定位目标商品。
  await loadProducts(productName)

  let matchedProduct = productOptions.value.find((item) => Number(item.id) === productId)

  // 兜底：若按名称搜索未命中（商品名在跳转期间被改、含特殊字符无法精确匹配等情况），
  // 退化为无关键词加载店铺下前 100 条商品再尝试一次。
  if (!matchedProduct) {
    await loadProducts('')
    matchedProduct = productOptions.value.find((item) => Number(item.id) === productId)
  }

  if (matchedProduct) {
    taskConfig.productId = productId
  } else {
    ElMessage.warning('未在当前店铺下找到该商品，请在商品下拉中手动搜索')
  }

  activeStep.value = 0
  return true
}
```

要点：

- 调用顺序严格按 "平台 → 店铺 → 商品" 完成，和人工操作 `taskConfig` 的依赖链一致。
- 任何一步失败都 `return true` 提前结束，避免继续污染 `taskConfig`，同时阻止 `onMounted` 里"单平台自动选中"兜底覆盖路由预填结果。
- 商品匹配使用双层兜底（关键词搜索 → 无关键词前 100 条），**完全避开 `loadProducts(String(productId))` 这种把 id 当关键词的签名陷阱**。

## 异常处理方案

### 商品没有平台

处理：

- 跳转时仍带 `productId`。
- 智能定价页提示“未找到商品所属平台，请手动选择平台后继续定价”。

### 商品没有 shopId

处理：

- 如果平台下只有一个店铺，自动选中该店铺。
- 如果平台下有多个店铺，提示用户选择店铺。

### 商品不在当前店铺下

处理：

- 先用 `loadProducts(productName)` 按商品名搜索，再在返回结果中按 `id` 精确匹配。
- 未命中时退化为 `loadProducts('')` 按店铺默认加载前 100 条商品再匹配一次。
- 两次都未命中，提示“未在当前店铺下找到该商品，请在商品下拉中手动搜索”。
- 页面停留在配置页，用户可手动修正。

### 商品名称缺失或为乱码

处理：

- 若 URL 里 `productName` 为空（例如用户手动拼装的链接），`loadProducts('')` 直接走默认候选集，由 id 精确匹配兜底。
- 若名称含特殊字符导致后端关键词模糊搜索不命中，同样回落到无关键词加载兜底。
- 整条预填流程不会因 `productName` 异常而中断。

### 用户未配置模型 API Key

处理：

- 智能定价页现有 `hasLlmConfig` 逻辑继续生效。
- “启动任务”按钮仍然禁用。
- 不需要在商品列表页做额外判断。

## 最终建议

建议采用“新增 `shopId` 响应字段 + 商品列表按钮跳转（携带 `productName`）+ 智能定价页按名称搜索 + id 精确匹配预填”的完整方案。

该方案改动清晰、兼容现有接口和前端流程，并且能处理同平台多店铺、店铺 SKU 超过 100 条、商品名含特殊字符等真实使用场景。相比只做前端跳转，它不会在多店铺数据下产生错误预填；相比直接创建任务，它不会绕过现有配置页和用户确认流程；相比把 `productId` 作为关键词传入 `loadProducts` 的朴素做法，它**避开了 `loadProducts(keyword)` 只支持名称模糊搜索的签名陷阱**，在不新增任何后端接口的前提下实现稳定的商品定位。
